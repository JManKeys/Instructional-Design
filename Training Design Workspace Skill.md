# Instructional Design: Training Design Package Generator

This skill guides the generation of a performance-based Training Design Package from source materials provided by the instructional designer. It is used when source documents need to be synthesized into curriculum anchored in critical job performance.

This skill is not a content summarizer. It is a design tool. Its outputs are governed by performance-based design principles and produce only what can be observed, practiced, and assessed under realistic conditions.

---

## SECTION 1 — PURPOSE

This skill produces Training Design Packages for instructional designers working across industries and client types. It applies consistently to the vast majority of client engagements where the goal is performance improvement — not knowledge transfer for its own sake.

The skill is a living document. It will be refined as real engagements reveal gaps or opportunities for improvement.

---

## SECTION 2 — REQUIRED INPUTS

Before beginning any work, confirm that the following inputs are available. If any required input is missing or unclear, stop and ask the designer before proceeding. Do not assume.

### Required

**1. Source Materials**
One or more documents provided by the instructional designer. May include any combination of: SME interview transcripts, existing training materials, technical manuals, regulatory standards, job descriptions, or process documentation. Accepted formats: PDF, Word, PowerPoint.

**2. Learner Profile**
A description of the target audience including: their role, current experience level, and any known performance gaps or challenges. If the instructor has described a specific class makeup, include that description here.

**3. Performance Context**
What the learner must be able to do on the job — the real conditions, constraints, and consequences they will face. If not explicitly stated in source materials, extract and present options for the designer to confirm before proceeding.

**4. Scope of Deliverables**
Which outputs are needed for this engagement. If not specified, present the full deliverables menu and ask the designer to select:

- Course outline / curriculum map
- Performance-based learning objectives
- Knowledge check *(written assessment)*
- Skills demonstration observation form
- Performance assessment rubric
- Classroom exercise handout
- Job aid / quick reference card
- Scenario-based practice activity
- Classroom slide deck with facilitator notes
- Facilitator guide
- SME validation summary

**5. Rubric Rating Scale** *(required if rubric is in scope)*
Designer specifies: 3-point (Exceeds / Meets / Needs Improvement) or 4-point (Exemplary / Proficient / Developing / Insufficient). Default to 3-point if not specified, but confirm with designer.

### Optional

**6. Client Brand Standards**
Any brand guide, template, or style specification provided by the client. If not provided, use a clean professional default and note this to the designer.

> **Designer Note:** The inputs list above is also your client intake framework — these are the questions that must be answered before any engagement begins.

## SECTION 3 — GOVERNING PRINCIPLES

### Layer 1: Four Questions — Ask These First, Before Any Output Is Generated

Before producing any deliverable, work through these four questions using the source materials provided. Where answers are unclear or absent, present your best interpretation and ask the designer to confirm before proceeding.

**1. What must people DO?**
Extract every observable task, decision, and consequence from the materials. Translate knowledge-dump content into performance terms. Create a task list. Flag content that describes what people must *know* rather than what they must *do* — treat it separately.

**2. How will we know they can do it?**
Define observable evidence of competence for each task. If it cannot be observed, it cannot be assessed. If it cannot be assessed, it does not belong in training.

**3. What practice will make that likely?**
Determine the type of practice that prepares the learner for real conditions. Match practice to the complexity, variability, and consequence of the actual job.

**4. What should NOT be trained at all?**
Identify content that belongs in a job aid, reference document, or workflow support rather than in training. Flag this explicitly in every output package.

---

### Layer 2: Explicit Design Rules — Apply These When Making Decisions

These rules govern specific design choices. They are not guidelines — they are decision logic.

- If task risk is HIGH AND consequence is SEVERE → require observable assessment, not self-report or completion-based check.

- If task frequency is LOW BUT consequence is HIGH → include scenario-based rehearsal.

- If cognitive load is HIGH → chunk content into manageable segments. Eliminate non-essential information before adding structure.

- If the transfer environment is VARIABLE → include a job aid. Do not rely on memory for information that changes by context.

- If a visual would support comprehension or performance → insert a placeholder. Do not leave visual decisions to production. Every placeholder must specify: (1) the type of visual needed, (2) the recommended source (extracted from provided materials / SME-provided required / licensed stock), and (3) a brief description of what the image must show. Never pull images from the internet without explicit designer approval. Never generate technical or procedural visuals — flag these as SME-provided required.

- If a visual EXISTS in the provided source materials → reference it by document name and location. Note it for extraction rather than recreating it.

- If a visual depicts a SAFETY-CRITICAL procedure, component, or condition → mark it as SME-provided required regardless of what other sources are available. Accuracy is non-negotiable for safety-relevant content.

---

### Layer 3: Design Principles — These Govern Every Output

**Principle 1 — Anchor design in critical job performance.**
Design for what people must do under real conditions, not what they must know. Training begins with tasks, decisions, and consequences.

**Principle 2 — Define observable standards of performance.**
Every objective is written as a visible behavior with clear criteria. If it cannot be observed, it cannot be assessed or improved.

**Principle 3 — Prioritize by operational risk and impact.**
Time and design effort are allocated based on consequence and frequency. High-risk tasks receive deeper practice and stronger assessment.

**Principle 4 — Align objectives, practice, and assessment with discipline.**
Learners practice exactly what they will be expected to perform. Assessment mirrors job conditions as closely as constraints allow.

**Principle 5 — Design for transfer under realistic conditions.**
Practice reflects the complexity, constraints, and variability of the work environment. Train judgment, not just recall.

**Principle 6 — Manage cognitive load intentionally.**
Eliminate non-essential information, sequence complexity deliberately, and separate explanation from application. Clarity beats volume.

**Principle 7 — Separate information from capability.**
Exposure to content is not equated with skill. When memory is not the goal, design job aids and workflow supports instead of overloading training.

**Principle 8 — Define measurable success before development begins.**
Agree on how performance improvement will be evaluated before assets are built. Completion metrics are not evidence of effectiveness.

**Principle 9 — Design for motivated engagement, not just performance compliance.**
Every design decision should consider how it supports or undermines the learner's sense of autonomy (meaningful choice), competence (achievable challenge with visible progress), and relatedness (connection to real work and real colleagues). Content that is technically correct but motivationally inert will not transfer. Where possible, design feedback and assessment as shared standards that instructor and learner work toward together — not as instruments of evaluation applied to the learner from above.

---

### Checkpoint Behavior

At four points in the process, stop and present options rather than proceeding on your own judgment. Do not advance past a checkpoint without designer confirmation.

- **Checkpoint 1 — Objective Quality Review:** Present drafted objectives and ask the designer to confirm, modify, or reject before continuing.
- **Checkpoint 2 — Assessment Alignment Check:** Present the proposed assessment approach and confirm it mirrors the objectives before generating questions.
- **Checkpoint 3 — SME Validation Summary:** Produce a structured summary of key decisions and assumptions for designer review with the SME before finalizing.
- **Checkpoint 4 — Transfer Realism Check:** Present practice and job aid decisions and ask the designer to confirm they reflect real working conditions.

## SECTION 4 — PROCESS

### Stage 1 — Intake and Orientation

When source materials are provided, do the following before generating anything:

Read all provided materials in full. Do not skim or summarize prematurely. Note the format and quality of each source — whether it is a transcript, a manual, a standards document, or existing training material.

Identify the learner profile from the inputs provided. If experience level, role, or performance context is missing or vague, stop and ask before proceeding. Do not assume.

Identify the facilitator profile from the inputs provided. 
Ask the designer:

"Who will be delivering this training — an experienced 
trainer who may not be a subject matter expert, or a 
subject matter expert who has limited facilitation 
experience?"

This answer governs how the facilitator guide is written 
throughout. If no facilitator guide is in scope, skip 
this question.

Identify the scope of deliverables requested. If not specified, present the full deliverables menu and ask the designer to select.

Identify client brand standards if provided. If not provided, note that a clean professional default will be used and confirm with the designer.

If a facilitator guide is in scope, ask the designer 
the following two questions before proceeding. Do not 
generate any facilitator guide content until both 
answers are confirmed.

**Facilitator Guide Question 1 — Profile:**
"Who will be delivering this training?"

- Experienced trainer who may not be a subject 
  matter expert
- Subject matter expert with limited facilitation 
  experience

This answer governs the depth of scripting and 
support language throughout the entire facilitator 
guide.

**Facilitator Guide Question 2 — Sections needed:**
"Which sections of the facilitator guide do you 
need for this engagement? Select all that apply:"

- Lesson Overview and Learning Outcomes
- Required Materials
- Pre-Session Preparation Checklist
- Lesson Flow
- Assessment Guide
- After the Session
- Complete Guide (all sections)

Generate only the sections confirmed by the designer.

**Dependency check — apply before generating:**

If Lesson Flow is selected but Required Materials 
is not — flag this to the designer before proceeding:
"Lesson Flow references Required Materials directly. 
I recommend including Required Materials to ensure 
the Lesson Flow is complete. Would you like to add 
it?"

If Assessment Guide is selected but neither the 
knowledge check nor the observation form is in 
the deliverables scope for this engagement — flag 
this to the designer before proceeding:
"The Assessment Guide references the knowledge 
check and observation form. Neither appears to be 
in scope for this engagement. Would you like to 
add them, or should I generate the Assessment 
Guide based on the objectives only?"

If Pre-Session Preparation Checklist is selected 
but Required Materials is not — flag this to the 
designer before proceeding:
"The Pre-Session Preparation Checklist is built 
from the Required Materials list. I recommend 
including Required Materials to ensure the 
checklist is complete. Would you like to add it?"

Do not proceed past these dependency checks without 
designer confirmation. Present all applicable flags 
at once rather than one at a time.

### Stage 2 — Content Analysis: The Four Questions

Working from the source materials, work through each of the four governing questions in order. This is analytical work — no outputs are generated yet.

**What must people DO?**
Extract every observable task, decision, and consequence from the materials. Translate knowledge-dump content into performance terms. Create a task list.

**How will we know they can do it?**
For each task on the list, define what observable evidence of competence looks like. Flag any task where evidence cannot be defined — these may not belong in training.

**What practice will make that likely?**
For each task, apply the explicit design rules from Layer 2 to determine the appropriate practice and assessment type.

**What should NOT be trained at all?**
Review the full content set and flag anything that belongs in a job aid, reference document, or SME resource rather than in training. This list appears explicitly in every output package.

---

### Checkpoint 1 — Objective Quality Review

Present the drafted task list and proposed objectives to the designer. For each objective show: the observable task, the condition, the standard, the risk rating, and the proposed assessment mode.

Designer confirms, modifies, or rejects each objective before proceeding to Stage 3. Do not move forward until this checkpoint is complete.

---

### Stage 3 — Assessment Design

Using the confirmed objectives from Checkpoint 1, design the assessment approach for each deliverable in scope.

**Knowledge checks:** Determine question type based on design rules. High-consequence tasks get scenario-based questions requiring judgment, not definition recall. Draft questions with answer rationale included — not just correct answers.

**Observation forms:** Define performance items that directly mirror the confirmed objectives. Assign criticality ratings. Draft observable standards for each item — not checklists of steps, but descriptions of what competent performance looks like.

**Rubrics:** Apply the rating scale specified in the inputs. Write behavioral descriptors for each criterion at each performance level. Descriptors must reflect actual task standards, not generic language.

**Classroom exercises:** Design the activity around the practice type determined in Stage 2. The exercise must require the learner to perform the task — not just read about it or answer questions about it in isolation.

**Job aids:** Identify the specific decision points or reference needs that triggered the job aid rule in Stage 2. Design the job aid to support performance in the moment — not to teach, to do.

---

### Checkpoint 2 — Assessment Alignment Check

Present the proposed assessment approach for each deliverable and confirm it directly mirrors the confirmed objectives. For each assessment item, note which objective it addresses.

Designer confirms alignment before proceeding to generation. Any assessment item that cannot be linked to a confirmed objective is removed or flagged for discussion.

---

### Stage 4 — Visual Placeholder Identification

Before drafting final outputs, review the full content set and assessment design for visual opportunities. For each identified location, apply the visual decision rules from Layer 2 and insert a placeholder specifying: visual type, recommended source, and description of what the image must show.

Safety-critical visuals are flagged as SME-provided required at this stage regardless of other available sources.

---

### Stage 5 — Output Generation

Generate all deliverables in scope using the confirmed objectives, assessment design, and visual placeholders. Outputs are produced in the order requested or in logical sequence if no order is specified.

Each output includes: the deliverable itself, a brief design rationale explaining key decisions made, the "what should not be trained" list with recommended treatment for each item, and all visual placeholders with full specifications.

**SDT Quality Check — Apply to Each Deliverable Before Finalizing**

Before completing any deliverable, review it against the three SDT needs. This is not a separate output — it is an internal check that improves the deliverable itself. If any need is poorly supported, revise before presenting to the designer.

- *Autonomy:* Does this deliverable give the learner meaningful choice at any point — in how they approach the task, demonstrate competence, or engage with the content? If every element is prescribed and controlling, flag it and consider where choice can be introduced without compromising the performance standard.

- *Competence:* Is the challenge level appropriate for the learner profile specified in the inputs? Does the learner have a clear picture of what success looks like and a realistic path to achieving it? If the standard is clear but the path is not, revise.

- *Relatedness:* Does the scenario, exercise, or assessment connect to work the learner actually recognizes? Are the conditions realistic enough that the learner can see themselves in the situation? If the context feels generic or abstract, revise to ground it in the real work environment.

- *Instructor-Learner Dynamic:* Where the deliverable involves instructor observation or feedback — particularly observation forms and rubrics — confirm that the language frames assessment as a shared standard rather than a judgment applied from above. Revise any language that positions the instructor as evaluator and the learner as subject.

---

### Checkpoint 3 — SME Validation Summary

Produce a structured summary of all key design decisions and assumptions made during the process. This document is intended for the designer to review with the SME before finalizing any deliverable.

The summary includes: the task list and its source in the materials, any content that was translated from knowledge-dump to performance terms, any content flagged as "should not be trained" and why, and any assumptions made where source materials were ambiguous.

---

### Checkpoint 4 — Transfer Realism Check

Present the practice design and job aid decisions and ask the designer to confirm they reflect real working conditions. Specifically: does the practice reflect the actual complexity and variability of the job? Do the job aids support the real decision points workers face? Are the assessment conditions realistic?

Designer confirms or requests revisions before the package is considered complete.

---

### Stage 6 — Final Package Assembly

Assemble all confirmed deliverables into a complete Training Design Package in the order specified in Section 5.10. Designer specifies which elements to include based on the scope confirmed in Stage 1.

---

## SECTION 5 — OUTPUT SPECIFICATIONS

This section defines the required structure and format for each deliverable. Follow these specifications for every output regardless of the scope of the engagement. Partial deliverables — when only some elements are requested — still follow the full specification for the elements included.

---

### 5.1 — What Should Not Be Trained List

This appears at the beginning of every Training Design Package, before any other deliverable. It is never optional.

**Structure:** A table with three columns: Content Item | Rationale for Exclusion | Recommended Treatment

- Rationale must reference a specific design principle or decision rule — not general judgment
- Recommended Treatment specifies one of: job aid, quick reference card, SME resource, pre-work reading, or workflow support
- If nothing from the source materials is excluded, state this explicitly with a brief rationale — do not simply omit the section

---

### 5.2 — Learning Objectives

**Structure:** A table with six columns for each objective:

| # | Observable Task | Condition | Standard | Risk Rating | Assessment Mode |

**Observable task:** Begins with an action verb describing visible behavior. Never uses: understand, know, appreciate, be aware of, or be familiar with.

**Condition:** Describes the real-world context in which the task must be performed — tools available, constraints present, environment characteristics.

**Standard:** Defines what acceptable performance looks like in measurable or observable terms. Includes what must be present and what errors are not acceptable.

**Risk rating:** High, Medium, or Low — determined by applying the consequence and frequency rules from Layer 2.

**Assessment mode:** Determined by risk rating and design rules. High risk always specifies observable assessment. Never defaults to multiple choice for high-risk tasks.

Minimum of one objective per major task cluster identified in Stage 2. If the objective list exceeds what can be reasonably addressed in the available instructional time, flag this to the designer before proceeding.

---

### 5.3 — Knowledge Check

**Structure:** Scenario-based questions grouped by topic cluster. Each question block contains: the scenario, the question, answer options, the correct answer with full rationale, and the common error explanation.

**Question design rules:**
- Mid-training and above: minimum 70% scenario-based questions requiring judgment or application. Maximum 30% recall or recognition questions.
- Early training: up to 50% recall questions permitted, but scenarios must still be included.
- Every high-risk objective must have at least one associated question.
- Answer rationale is not optional — it is where the instructional value lives. Write rationale that explains why the correct answer is correct AND why each incorrect option is wrong or incomplete.
- Distractors must be plausible — drawn from common errors or misconceptions, not obviously wrong.

---

### 5.4 — Skills Demonstration Observation Form

**Structure:** Organized in parts that mirror the major task clusters from the learning objectives. Each part contains a table with four columns:

| ✓ | Performance Item | Observable Standard | Result |

**Observable standard:** Describes what competent performance looks like in practice — not a list of steps, but a description of the behavior and its quality. Written so that two different observers would reach the same conclusion when watching the same performance.

**Result rating:** S (Satisfactory) / U (Unsatisfactory) / N/O (Not Observed)

**Safety-critical items:** Marked with ★. Any U rating on a ★ item means the session is incomplete regardless of other ratings. State this explicitly on the form.

**Form includes:** Student name, instructor name, date, session number, rating key, and a formal instructor sign-off block with space for notes.

**SDT note:** Language throughout the form frames assessment as shared standards — not as judgment applied from above. Avoid language that positions the instructor as evaluator and the learner as subject.

---

### 5.5 — Performance Rubric

**Structure:** A table with criteria in the left column and performance levels across the top. Rating scale is specified in the inputs — default is 3-point (Exceeds Expectations / Meets Expectations / Needs Improvement) unless designer specifies otherwise.

**Criteria:** Drawn directly from the performance items in the observation form. Rubric and observation form must be aligned — if an item appears in one it appears in the other.

**Descriptors:** Written in behavioral terms specific to the task. Generic language — "demonstrates understanding," "shows effort," "participates actively" — is not acceptable. Each descriptor must describe what the performance actually looks like at that level for that specific task.

**Point values:** Assigned to each level and stated clearly. 
Total possible score = number of rubric criteria × highest point 
value. Minimum passing score included if designer specifies a 
threshold. Rubric criteria count must match the number of 
distinct criteria rows in the rubric table — not the number 
of observation form items.

**SDT note:** Rubric language frames each level as a point on a development continuum, not a judgment of the person. The highest level describes mastery worth working toward, not perfection that is rarely achieved.

---

### 5.6 — Classroom Exercise Handout

**Structure:** A standalone document the learner works through during or after instruction. Contains: exercise title, learning objective it addresses, scenario or context, task instructions, workspace for learner responses, and a debrief guide for the instructor.

**Exercise design rules:**
- The exercise must require the learner to perform the task — not read about it, watch it, or answer questions about it in isolation.
- Scenario must reflect realistic conditions from the transfer environment — not simplified or sanitized versions.
- Debrief guide is not optional. It tells the instructor what to look for, what common errors to address, and how to connect the exercise back to the job.

**SDT note:** Where possible, build meaningful choice into the exercise. This may be choice of approach, choice of scenario variant, or choice of how to demonstrate the outcome. Choice must not compromise the performance standard.

---

### 5.7 — Job Aid

**Structure:** Determined by the decision point it supports. Format options: decision tree, quick reference card, checklist, or annotated diagram. Select the format based on the nature of the task and state the rationale for the choice.

**Job aid design rules:**
- A job aid replaces memory — it must be usable in the moment without training or explanation.
- Language is action-oriented and imperative — tells the worker what to do, not what to know.
- If the job aid requires more than one page to be usable in the field, flag this to the designer. Job aids that are too long will not be used.
- Visual placeholders follow the rules from Layer 2. Technical diagrams are marked SME-provided required.

---

### 5.8 — Visual Placeholders

Every visual placeholder — regardless of which deliverable it appears in — follows this exact format:

> **[VISUAL PLACEHOLDER — Type:** *(diagram / photograph / illustration / chart)* **— Source:** *(extract from provided materials / SME-provided required / licensed stock)* **— Description:** *(specific description of what the image must show, including any technical accuracy requirements)* **— Notes:** *(any safety-critical flags or replacement instructions)***]**

Placeholders are never left vague. "Insert image here" is not acceptable. Every placeholder gives the production team a complete brief.

---

### 5.9 — SME Validation Summary

**Structure:** A document organized in four sections:
1. Confirmed task list with source references
2. Content translated from knowledge-dump to performance terms
3. Content excluded from training with rationale
4. Assumptions made where source materials were ambiguous

**Audience:** Written for the SME — not the designer. Language is plain, jargon-free, and explains design decisions in terms of job performance rather than instructional design theory. The SME should be able to read it and confirm or correct without needing an ID background.

---

### 5.10 — Package Assembly Order

When delivering a complete Training Design Package, assemble deliverables in this order:

1. What Should Not Be Trained
2. Learning Objectives
3. Classroom Slide Deck with Facilitator Notes
4. Facilitator Guide
5. Knowledge Check
6. Skills Demonstration Observation Form
7. Performance Rubric
8. Classroom Exercise Handout
9. Job Aids
10. SME validation summary

If only a subset of deliverables is requested, the order above still applies to the elements included.

### 5.11 — Classroom Slide Deck with Facilitator Notes

**Structure:** A slide-by-slide outline designed for classroom delivery. 
Each slide entry contains four elements: slide title, main content 
points (maximum 5 per slide), facilitator notes, and any visual 
placeholder following the rules in 5.8.

**Slide design rules:**
- Each slide covers one idea. If a slide requires more than 5 content 
  points, split it into two slides.
- Content points are written as brief, scannable phrases — not full 
  sentences. Full explanation lives in the facilitator notes, not on 
  the slide.
- Facilitator notes are written in plain conversational language — 
  what the instructor says and does, not what the slide shows. Notes 
  include: key points to emphasize, common student questions to 
  anticipate, connections to hands-on demonstration components, and 
  any safety-critical points requiring special emphasis.
- Facilitator notes must be written exclusively for the 
  instructor delivering the session. Do not include internal 
  design rationale, ACS code references, skill methodology 
  notes, or any language referencing the design process. 
  If a connection to a specific task or standard is relevant 
  for the facilitator, express it in plain instructional 
  language — not as a reference to source documents or 
  design decisions.
- Slides follow the sequence of confirmed learning objectives from 
  Checkpoint 1. Every objective must be addressed by at least one 
  slide cluster.
- On scenario-based discussion slides, the scenario context 
  is presented as a paragraph or short prose block — not as 
  bullets. Questions that follow the scenario are presented 
  as bullets underneath. This visually distinguishes "here 
  is the situation" from "here is what to discuss."
- Slide count and estimated delivery time information must 
  appear in the facilitator notes of slide 1 — not at the 
  end of the deck. This gives the facilitator a complete 
  picture of the session before they begin delivering it.
- If a facilitator guide is included in the deliverables 
  scope for this engagement, do not include slide count, 
  estimated delivery time, or session timing information 
  in the slide notes at all. That information belongs 
  exclusively in the facilitator guide. If no facilitator 
  guide is in scope, include it in the slide 1 notes only.
- This output is structured for import into a branded PowerPoint 
  template. Do not prescribe visual design — focus on content 
  structure and facilitator guidance.
- Cluster labels (Cluster A, Cluster B, etc.) are used 
  internally to organize content but must not appear on 
  slides. Section transitions are communicated through 
  slide design and content flow — not by labeling the 
  cluster name explicitly.

**SDT note:** Build in at least one discussion question or brief 
activity prompt per major topic cluster. These give learners 
voice and choice within the classroom session and support 
the autonomy and relatedness needs.

### 5.12 — Facilitator Guide

**Before generating, confirm the facilitator profile 
from Stage 1 intake.** The profile — experienced trainer 
or SME facilitator — governs the depth of scripting and 
support language throughout this document.

**Audience:** Written for the person delivering the 
session. Language is plain, direct, and actionable. 
Assumes subject matter expertise OR facilitation 
expertise — not necessarily both.

---

#### PART 1 — LESSON OVERVIEW AND LEARNING OUTCOMES

A single combined opening section that gives the 
facilitator a complete picture of the session before 
they open a single slide.

**Lesson Overview:** 2–3 sentences describing the main 
point of the lesson and why it matters to the learner's 
job. Written in plain language — not instructional 
design language. The facilitator should be able to read 
this aloud as an opening statement if needed.

**Key Points:** 3–5 bullet points summarizing the most 
important content themes of the lesson.

**Learning Outcomes:** The confirmed learning objectives 
from Checkpoint 1, written in plain language the 
facilitator can communicate directly to students at the 
start of the session. Framed as: "By the end of this 
session, you will be able to..."

---

#### PART 2 — REQUIRED MATERIALS

A complete list of everything the facilitator needs to 
successfully deliver the session. Organized in three 
categories:

**Digital materials:**
List all files required — slide deck, knowledge check, 
observation form, rubric, classroom exercise handout, 
job aids. Include file names where known.

**Printed materials:**
List all documents that must be printed and how many 
copies are needed per student and per facilitator.

**Physical materials:**
List all physical items needed for the session — 
training aids, parts, components, tools, equipment, 
and any demonstration items. Be specific: "one 
serviceable brake assembly per student pair" is more 
useful than "brake components."

**Technology and room setup:**
Projector or display, any test equipment needed for 
demonstration, and any specific room configuration 
requirements.

---

#### PART 3 — PRE-SESSION PREPARATION CHECKLIST

Formatted as a checklist the facilitator works through 
before the session begins. Not prose — every item is 
a checkbox.

**Standing items that appear on every checklist 
regardless of course content:**

☐ Safety briefing prepared — review safety requirements 
specific to any hands-on components of this session 
before students arrive. For sessions involving 
pressurized systems, jacking operations, or live 
aircraft components, confirm all required PPE is 
available and all safety procedures will be explicitly 
covered before any hands-on work begins.

☐ All digital files confirmed accessible and 
displaying correctly on the delivery system

☐ All printed materials copied in correct quantities

☐ All physical training aids and components confirmed 
present and in correct condition for demonstration

☐ Room configured per session requirements

☐ Any guest speakers or additional SMEs confirmed

☐ Student roster confirmed — note any students 
requiring accommodation or with known experience gaps

**Course-specific checklist items** are generated based 
on the required materials identified in Part 2 and the 
lesson flow in Part 3.

---

#### PART 4 — LESSON FLOW

The architecture of the session from opening to close. 
Organized as a timed sequence of segments.

**Format for each segment:**

| Time | Segment | Type | Duration | Materials |
|------|---------|------|----------|-----------|

Types: Opening / Instruction / Discussion Activity / 
Knowledge Check / Hands-On Demonstration / Skills 
Observation / Debrief / Close

**Calibration by facilitator profile:**

*Experienced trainer profile:* Each segment includes 
timing guidance and transition cues. Delivery detail 
lives in the slide notes. The lesson flow tells the 
facilitator what happens when and for how long — 
not what to say word for word.

*SME facilitator profile:* Each segment includes 
timing guidance AND suggested language for: opening 
the session, transitioning between topics, launching 
and debriefing discussion activities, handling 
unexpected student questions, and closing the session. 
Language is suggested, not prescribed — framed as 
"you might say..." rather than a script to be read 
verbatim. The goal is confidence, not performance.

**Both profiles include:**
- Explicit transition point from classroom instruction 
  to hands-on observation — what the facilitator says 
  and does to make that shift
- Discussion activity timing and debrief guidance 
  for each activity
- Knowledge check administration point — when it 
  is handed out, how long students have, what 
  facilitator does during that time
- Differentiation note: what to do if some students 
  are ahead and others are struggling — specific 
  to this content, not generic advice

---

#### PART 5 — ASSESSMENT GUIDE

How the facilitator administers and interprets both 
assessment components of the course.

**Knowledge Check Administration:**
When to distribute, how long to allow, whether 
open-book or closed-book, how to score, and when 
to return results to students. Include: what the 
facilitator does if a student does not meet the 
passing threshold and what re-assessment looks like.

**Skills Observation Administration:**
How to set up the observation, what the facilitator 
is watching for at each station or task, how to 
use the observation form during live performance, 
and how to complete the sign-off block. Include:

- How to handle a safety-critical (★) item that 
  results in a U rating — what to say, what to 
  document, and what happens next
- How to give feedback after the observation in 
  language that frames the standard as shared — 
  not as a judgment delivered from above
- Re-observation procedure: what must be repeated, 
  what documentation is required

**What Success Looks Like:**
A plain-language summary of what a strong performance 
looks like across both assessment components — written 
so the facilitator can describe the standard to 
students before they are assessed, not after.

#### PART 6 — AFTER THE SESSION

What the facilitator does when instruction ends.

**Observation form handling:** Where completed forms 
go, who receives copies, and how they are stored or 
filed per program requirements.

**Performance documentation:** How to record student 
results in the program tracking system or logbook. 
What a satisfactory session entry looks like versus 
an incomplete or unsatisfactory entry.

**Students who did not meet standard:** What happens 
next for a student with an incomplete or unsatisfactory 
session — who is notified, what is scheduled, and 
what documentation travels with the student to the 
re-observation.

**Handoff to next session:** Any information the 
next facilitator or phase needs to know based on 
what happened in this session — particularly for 
students who are progressing with conditions or 
flagged performance areas.

**Facilitator debrief note:** A prompt reminding 
the facilitator to note anything that should improve 
future deliveries — content gaps, student questions 
that weren't covered, physical materials that didn't 
work as expected. This note stays with the designer, 
not the student record.

### 5.13 — PowerPoint Slide Deck (Unbranded)

**Structure:** A .pptx file generated directly from the classroom 
slide deck content in Section 5.11. Each slide in the outline 
becomes a slide in the presentation. Facilitator notes are placed 
in the notes panel of the corresponding slide — not on the slide 
itself.

**Generation rules:**
- Slide title maps to the slide title field
- Content points appear as the slide body text — one point per 
  line, clean and scannable
- Facilitator notes go in the notes panel in full — do not 
  truncate
- Visual placeholders appear as a text box on the slide noting 
  the placeholder description
- Visual placeholder text boxes must contain the complete 
  placeholder specification — type, source, full description, 
  and all notes. Do not truncate under any circumstances. 
  If the text exceeds the visible text box area, reduce the 
  font size to fit rather than cutting the text. The production 
  team depends on the complete brief to source or create the 
  correct image.
- No client branding applied — clean professional default 
  formatting only
- This output is intended as a working draft for review and 
  import into a branded template

**How to trigger:** After the full training package is confirmed, 
generate the PowerPoint by converting the Section 5.11 slide 
deck outline into a .pptx file and provide it as a downloadable 
file.
